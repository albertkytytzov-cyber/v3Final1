# PERFORM Constructor Core Stack

Версия: черновик v1, 2026-06-05

Назначение: зафиксировать ядро конструктора подготовки, чтобы PERFORM строил планы не из свободного текста, а из входных данных, тестов, фазы, рисков, доказательной базы и объяснимых правил.

Этот документ описывает продуктовую и техническую основу режима `Собрать план подготовки`. Код приложения должен внедряться только после утверждения этой структуры.

## 1. Главный принцип

Конструктор не должен “генерировать красивый календарь”. Он должен строить план по цепочке:

```text
соревнование
+ спортсмен
+ фаза подготовки
+ цели
+ тесты / исходные точки
+ текущее состояние
+ ограничения и риски
+ доказательная база
= редактируемый план + объяснение "почему так"
```

Если в цепочке не хватает данных, система не должна молча додумывать. Она должна:

- снизить уверенность;
- показать, каких данных не хватает;
- предложить безопасный вариант;
- запросить тест или тренерское подтверждение;
- не назначать рискованный блок автоматически.

## 2. Рабочая структура экрана

Утверждаем базовую логику конструктора:

```text
левая колонка: ввод данных
центральная колонка: что система поняла
правая колонка: черновик плана и объяснение
```

### Левая колонка: ввод данных

Содержит пошаговый ввод:

1. Соревнование.
2. Спортсмен.
3. Цели подготовки.
4. Тесты и исходные точки.
5. Ограничения.
6. Утверждение.

### Центральная колонка: интерпретация

Система показывает:

- главную задачу;
- почему выбраны эти цели;
- какие тесты подтверждают или ограничивают цель;
- какие риски есть;
- насколько уверенный вывод;
- какие данные нужно уточнить.

### Правая колонка: черновик плана

Система показывает:

- недельную структуру;
- дни и блоки;
- типы нагрузки;
- восстановление;
- контрольные точки;
- блок `Почему так`;
- что тренер может отредактировать.

### 2.1 Подключение в Planning Studio

Текущий рабочий путь в интерфейсе тренера:

```text
Planning Studio
-> вкладка "Конструктор"
-> слева ввод данных
-> по центру "что система поняла"
-> справа draft-план
-> "Сохранить как шаблон"
```

Правила поведения:

- если у выбранного спортсмена есть календарь стартов, конструктор показывает эти соревнования в поле `Соревнование из календаря`;
- ближайший будущий старт может подставляться автоматически, но тренер может выбрать другой старт или перейти в ручной режим;
- обычный экран не должен дублировать поля старта, даты, приоритета и взвешивания: эти данные берутся из календаря;
- ручные поля старта показываются как резервный раскрываемый блок, когда старта нет в календаре или нужен временный черновик;
- дата старта используется как календарный якорь: система считает, сколько дней осталось до соревнования, и рекомендует фазу подготовки;
- при изменении даты вручную конструктор пересчитывает фазу и длину чернового цикла, чтобы не оставлять старую фазу от другого старта;
- блок `Что система поняла` обязан объяснять решение через календарь, сезон, олимпийский цикл, фазу и точные дни до старта;
- если до старта осталось 0-4 дня, система не должна писать про “развивающий план”: это стартовое окно, где уверенность может быть снижена из-за нехватки данных, но логика остаётся подводящей и безопасной;
- для 0-4 дней до старта автофокус не включает ЛМВ, плотность борьбы, силу, интервальную работу или развитие скорости;
- стартовое окно не может строить `2 тренировки x каждый день`: максимум одна короткая сессия в день, часть дней должна быть восстановительной/без ковра;
- конструктор не назначает план спортсмену сразу;
- сначала создаётся объяснимый draft-план;
- тренер проверяет цели, риски, нехватку данных и структуру недель;
- после подтверждения draft сохраняется как обычный шаблон;
- назначение спортсмену идёт уже через существующую логику шаблонов.

Техническая связка:

- frontend: вкладка `constructor` в `PlanningStudioView`;
- API: `POST /api/v1/plans/constructor/draft`;
- shared core: `buildPerformConstructorDraft(input)`;
- сохранение: `buildConstructorTemplatePayload(draft, name)` -> `POST /api/v1/plans/templates`.

Preview-режим может собирать draft локально через shared core, чтобы макет и логика были проверяемы без локальной базы. В рабочем режиме тренера генерация должна проходить через серверный API.

## 3. Core Data Model

### 3.0 Season Strategy Snapshot

Перед сборкой черновика конструктор должен получить не только выбранный старт, но и стратегический снимок сезона.

Назначение `SeasonStrategySnapshot`: превратить олимпийский цикл, сезон, стратегию года и календарь стартов в понятные ограничения для генератора.

Цепочка:

```text
олимпийский цикл
+ сезон и стратегия года
+ календарь стартов спортсмена
+ выбранный целевой старт
= SeasonStrategySnapshot
-> ограничения конструктора
-> draft-план
```

Snapshot фиксирует:

- номер года олимпийского цикла;
- этап года: база, углублённая специальная подготовка, квалификация/реализация, олимпийский пик;
- стратегию сезона: `single_peak`, `double_peak`, `multi_peak`;
- роль выбранного старта: главный пик, второй пик, квалификация, контрольный старт;
- точное количество дней до старта;
- фазу текущего окна;
- разрешённые режимы работы: развитие, поддержание, перенос, активация, восстановление;
- запрещённые режимы и фокусы;
- обязательные фокусы;
- недельный ритм и предупреждения.

Ключевое правило:

```text
Если до главного старта A осталось <= 30 дней,
конструктор строит ровно оставшиеся дни, а не фиксированные 30 дней.
Развитие запрещено; остаются поддержание, перенос, активация, восстановление.
```

Конструктор не должен округлять оставшиеся дни к старым шаблонам `7/10/14/21/30`. Эти числа могут оставаться быстрыми ручными ориентирами, но не управляют автоматическим планом от календаря.

Пример: до старта осталось `4 дня`.

```text
Фаза: start_window
Длина draft-плана: 4 дня
Фазовая карта: Д-4...старт
Нельзя: подставлять 7-дневный taper-шаблон как длину цикла
Нельзя: ставить 8 ковровых тренировок за 4 дня
Нельзя: автоматически выбирать ЛМВ/СФП ног как фокус
Нужно: построить 4 календарных дня стартового окна с короткой активацией, весом, сном и восстановлением
```

Пример: сегодня `08.06`, чемпионат Европы `06.07`, осталось `28 дней`.

```text
Год цикла: 2-й год, углублённая специальная подготовка
Стратегия сезона: multi_peak
Роль старта: main_peak
Фаза: special_preparation
Длина draft-плана: 28 дней
Запрещено: development
Фокус: специальная борьба, соревновательная модель, поддержание СФП, вес, восстановление, подводка
```

Это значит, что конструктор не должен предлагать “развитие скорости” или “развитие ЛМВ” как отдельные задачи. Он должен переводить их в поддержание резкости, перенос СФП в борьбу и контроль свежести.

### 3.1 Competition Profile

Обязательные поля:

```text
competitionId
name
level
priority
startDate
daysToStart
weighInDate
weightClass
competitionFormat
expectedBoutCount
location
timezone
travelRequired
climateContext
```

Источник данных:

- первичный источник: `CompetitionPlanSummary` выбранного спортсмена;
- fallback: ручной ввод старта в конструкторе;
- `daysToStart` считается от текущей даты до `startDate`;
- рекомендуемая фаза конструктора:
  - больше 60 дней: `base`;
  - 31-60 дней: `development`;
  - 15-30 дней до главного старта: предсоревновательный Europe-case стек, внутри `special_preparation`;
  - 5-11 дней до главного старта: интеграция, снижение объёма и суперкомпенсация, внутри `taper`;
  - 0-4 дня до старта: `start_window`;
  - после старта: `recovery`.

Значения:

| Поле | Смысл | Пример |
|---|---|---|
| `level` | уровень старта | Olympics, World, Europe, national, control |
| `priority` | важность | main, secondary, test |
| `competitionFormat` | формат | tournament_day, multi_day, camp_control |
| `expectedBoutCount` | ожидаемое число схваток | 3-5 |
| `travelRequired` | влияет ли перелёт | true/false |

### 3.3 Preparation Context

```text
currentPhase
targetPhase
cycleLengthDays
availableTrainingDays
sessionsPerDay
sessionsPerWeek
coachPriorityGoals
forbiddenGoals
phaseConstraints
```

Важно: `sessionsPerWeek` не является декоративным полем. Генератор должен использовать его при сборке черновика. Методические карточки ниже задают опорные дни и контрольные стимулы, но не являются полным недельным расписанием. Если тренер выбирает 5-6 тренировочных дней в неделю, черновик обязан расширить неделю техническими, восстановительными, аэробными или подводящими днями, а не возвращать только 2-3 контрольных дня карточки.

Фазы:

| Фаза | Когда используется | Что разрешено |
|---|---|---|
| `base` | 61+ день до старта | база, сила, ЛМВ, аэробная работа |
| `development` | 31-60 дней до старта | развитие качеств |
| `special_preparation` | 12-30 дней до главного старта | вход в блок, основной специальный микроцикл, перенос силы/СФП в борьбу |
| `taper` | 5-11 дней до главного старта | интеграция, снижение объёма, суперкомпенсация, короткая активация без развития |
| `start_window` | 0-4 дня до старта | свежесть, вес, техника, сон |
| `recovery` | после перегруза/старта | восстановление и контроль |

### 3.3.1 Europe-Case для главного старта до 30 дней

Если старт уровня `continental/world/olympics` или приоритет `A`, а до соревнования `<= 30` дней, конструктор не имеет права строить обычный развивающий блок. Это уже предсоревновательная подготовка по эталону `docs/europe-2026-plan-analysis.md`.

Правильная логика:

- фаза `development` автоматически нормализуется в `special_preparation`;
- 30 дней разбиваются на календарные этапы:
  - вход в предсоревновательный блок;
  - основной специальный микроцикл;
  - интеграция и снижение объёма;
  - подводка и пик;
  - стартовое окно;
- борьба и техника остаются ядром каждого активного тренировочного дня;
- ОФП/СФП, ЛМВ, хват, интервальная работа и силовой тонус являются поддержкой и переносом в борьбу, а не самостоятельным “развитием качества”;
- дни строятся как две отдельные сессии, если `sessionsPerDay = 2`;
- допустимы оба порядка:
  - `УТРО: техника/борьба -> ВЕЧЕР: ОФП/СФП`;
  - `УТРО: ОФП/СФП -> ВЕЧЕР: техника/борьба`;
- шестидневная тренировочная неделя не должна превращаться в `6 дней x 2 тренировки`;
- базовый ритм нагрузки для недели с воскресеньем отдыхом:
  - `ПН`: полный день;
  - `ВТ`: полный день;
  - `СР`: половинчатый разгрузочный день, одна тренировка;
  - `ЧТ`: полный день;
  - `ПТ`: полный день;
  - `СБ`: половинчатый разгрузочный день, одна тренировка;
  - `ВС`: отдых;
- половинчатый день означает одну короткую/контролируемую сессию и восстановительные процедуры вместо второй тренировки: сауна, мобилити, массаж, сон, питание, контроль веса и пульса;
- половинчатый день не должен автоматически ставить ковёр, технику или борьбу: в среду/субботу основной смысл - смена обстановки и снятие накопленного напряжения;
- допустимые разгрузочные варианты: кросс Z1-Z2, прогулка, поход, велосипед, бассейн, сауна, массаж, мобилити, дыхание;
- смена обстановки является частью подготовки, потому что спортсмен не должен психологически “жить на ковре” каждый день последнего месяца;
- каждый рабочий день должен иметь `Разминка -> основная работа -> Заминка` внутри каждой сессии;
- интервальная работа должна быть борцовской: например `резина -> проход -> клинч/давление -> защита/спрол`, а не абстрактные ускорения;
- в подводке короткая скорость используется только как активация/резкость, не как развивающая скоростная работа;
- рядом с активацией в последние дни обязательно остаются техника, вес, сон и восстановление.

Автовыбор целей для главного старта:

- в интерфейсе этот блок называется не `Цели цикла`, а `Фокус подготовки`;
- при `<= 30` днях система сама выбирает предсоревновательный стек: специальная борцовская работа, соревновательная модель, поддержание СФП, контроль веса, восстановление, качество подводки;
- режим `development` запрещён: каждый фокус получает режим `maintenance`, `transfer`, `activation` или `recovery`;
- `speed_first_action` в последнем месяце отображается как поддержание резкости первого действия, а не как развитие скорости;
- `legs_lme` отображается как поддержание СФП ног, а не как развитие ЛМВ;
- при `<= 23` днях фокус смещается к основному специальному микроциклу, интеграции и подводке;
- при `<= 14` днях развитие убирается, остаются техника, подводка, вес и восстановление;
- при `<= 7` днях остаются только свежесть, короткая активация, вес, сон и техническая уверенность;
- при `<= 4` днях ЛМВ, плотность борьбы, силовая, интервальная работа и развитие скорости блокируются даже если раньше были выбраны в форме;
- при `<= 4` днях каждый день имеет не больше одной короткой сессии, а восстановительные дни не должны автоматически получать ковёр или технику;
- тренер может добавить или убрать цели, но базовый стек должен подставляться автоматически, чтобы не заставлять тренера вручную выбирать очевидные блоки для последнего месяца.

Черновик такого плана должен быть похож не на ОФП-календарь, а на борцовский план подготовки: техника, ситуации, соревновательные циклы `3 мин + 30 сек + 3 мин`, борьба без силовой рубки, контроль веса, восстановление и подводка.

Календарная привязка:

- для главного старта план должен строиться по реальным датам, а не только по абстрактным `ПН/ВТ`;
- label дня должен показывать расстояние до старта и дату, например `Д-30 / ВС 07.06`;
- дата считается от `competition.startDate` назад на `cycleLengthDays`;
- если в `availableTrainingDays` воскресенье не выбрано, конструктор не создаёт тренировочный день на воскресенье, даже если календарный 30-дневный отрезок начинается с воскресенья;
- воскресенье в шестидневной неделе остаётся днём отдыха/семьи/психологического восстановления, а не скрытым тренировочным днём;
- это нужно, чтобы тренер сразу видел, что именно делать в конкретное число, включая дорогу, взвешивание, разгрузку и стартовое окно.

### 3.3.2 Протокол проверки изменений конструктора

Перед завершением любой правки конструктора нужно сделать не один, а минимум три прохода проверки:

1. Логический проход: проверить, какая фаза выбрана, какие фокусы разрешены/запрещены и почему.
2. Проход черновика: посмотреть фактически созданные дни, сессии, блоки, ковёр/не ковёр, нагрузку и восстановление.
3. Регрессионный проход: запустить тесты на ключевые сценарии `30 дней`, `28 дней`, `14 дней`, `4 дня`, импорт и сохранение шаблона.

Для стартового окна отдельные обязательные проверки:

- `4 дня` не округляются в `7`;
- нет ЛМВ/плотности борьбы/силовой/интервалов в автофокусе;
- нет схемы `4 дня x 2 тренировки`;
- есть восстановительные дни без ковра;
- объяснение говорит “стартовое окно”, а не “развивающий план”.

## 4. Test Profile

Каждая цель должна иметь тестовую опору. Если её нет, система не должна делать жёсткий план.

### 4.1 Скорость

| Тест | Что показывает | Когда нужен |
|---|---|---|
| 10 м | первый шаг, стартовое ускорение | цель `speed_first_action` |
| 20 м | короткое ускорение | скорость атаки/перемещения |
| 30 м | общий скоростной отрезок | если есть беговая подготовка |
| flying 10 м | чистая скорость без старта | продвинутая диагностика |
| реакция | старт по сигналу | борьба, первый контакт |

Правило:

```text
Если цель скорость, но нет 10 м и 20 м,
PERFORM строит только осторожный speed-maintenance блок или просит тест.
```

### 4.2 Взрывная сила

| Тест | Что показывает |
|---|---|
| прыжок вверх | мощность ног |
| прыжок в длину | горизонтальный взрыв |
| бросок медбола | перенос силы корпуса/рук |
| RFD / force plate | скорость развития усилия, если доступно |

Правило:

```text
Если скорость слабая, но взрыв нормальный,
система выбирает короткую скорость и реакцию, а не тяжёлую силовую.
```

### 4.3 Максимальная сила

| Тест | Что показывает |
|---|---|
| присед / half squat | сила ног |
| тяга | задняя цепь |
| жим | верх |
| подтягивания | относительная сила |
| 3RM / расчетный 1RM | безопаснее, чем обязательный 1RM |

Правило:

```text
Если до старта меньше 14 дней,
тяжёлая максимальная сила запрещена без фазы strengthMaintenance.
```

### 4.4 Локальная мышечная выносливость

| Зона | Тесты / признаки |
|---|---|
| ноги | статодинамика, удержания, качество входов |
| руки/хват | динамометрия, вис, канат, удержание партнёра |
| корпус/таз | удержания, мост, стабилизация, качество позиции |

Правило:

```text
Если локальная усталость ног высокая,
система не ставит спринты и тяжёлые входы в ноги рядом.
```

### 4.5 Аэробная база

| Тест | Что показывает |
|---|---|
| пульсовые зоны | контроль интенсивности |
| восстановление ЧСС | способность восстанавливаться |
| 12 мин / 3200 м | общий аэробный уровень |
| Yo-Yo / 30-15 IFT | интервальная выносливость |

Правило:

```text
Если RHR растёт и сон плохой,
аэробная работа может остаться, но интенсивность снижается.
```

### 4.6 Борцовская специальная работа

| Метрика | Что показывает |
|---|---|
| успешные входы | качество атаки |
| качество техники 1-5 | тренерская оценка |
| round structure | соответствие соревновательной модели |
| contact intensity | плотность контакта |
| fatigue skill score | качество под утомлением |

Правило:

```text
Если цель чистая техника,
перед ней нельзя ставить блок, который случайно ломает качество.
```

## 5. Readiness / State Layer

Система должна учитывать ежедневное состояние:

```text
readinessScore
sleepDuration
sleepQuality
rhr
hrv
weight
painZones
soreness
fatigue
motivation
illness
deviceDataConfidence
coachComment
athleteDiary
```

### Confidence

| Уровень | Условие | Поведение |
|---|---|---|
| high | есть тесты, readiness, сон, RHR, вес, дневник | можно строить уверенный план |
| medium | часть данных есть, но не хватает 1-2 важных блоков | строить осторожно, показать пробелы |
| low | нет тестов или нет состояния | только безопасный шаблон и запрос данных |

## 6. Risk Stack

Риски должны быть отдельным слоем, а не просто текстом ИИ.

### 6.1 Weight Risk

Сигналы:

- резкое снижение веса;
- вес выше цели близко к старту;
- плохой сон + снижение веса;
- рост RHR + снижение веса;
- нет данных гидратации/самочувствия.

Правило:

```text
Если weightRisk высокий,
PERFORM не усиливает гликолитику и контактную плотность.
```

### 6.2 Local Load Risk

Сигналы:

- боль в зоне нагрузки;
- локальная усталость ног/рук/спины;
- тяжёлый блок на ту же зону в последние 24-48 часов;
- падение качества техники.

Правило:

```text
Если зона перегружена,
следующий блок либо заменяется, либо переводится в технику/восстановление.
```

### 6.3 Recovery Risk

Сигналы:

- сон ниже baseline;
- RHR выше baseline;
- HRV ниже baseline;
- readiness низкая;
- болезнь/температура;
- высокая усталость.

Правило:

```text
Если recoveryRisk высокий,
система снижает интенсивность и не назначает новый развивающий стимул.
```

### 6.4 Competition Proximity Risk

Сигналы:

- до старта меньше 14 дней;
- до старта меньше 7 дней;
- взвешивание близко;
- перелёт/смена часового пояса.

Правило:

```text
Чем ближе старт, тем меньше права на развитие качества.
Последние дни: свежесть, техника, вес, сон, контроль.
```

## 7. Generation Rules

### 7.1 Правила по сроку до старта

| Дней до старта | Что можно делать | Что нельзя делать автоматически |
|---:|---|---|
| 60+ | база, сила, ЛМВ, аэробная работа, тесты | игнорировать долгосрочную фазу |
| 30-60 | развитие качеств, специальные блоки | ставить всё сразу без совместимости |
| 19-30 перед главным стартом | вход в предсоревновательный блок: Z2/ОФП/СФП, сила и хват с контролем | запускать абстрактную фазу развития |
| 12-18 перед главным стартом | основной специальный микроцикл: борьба, 2x3, контактная плотность, финишная способность | превращать неделю в один скоростной блок |
| 5-11 перед главным стартом | интеграция: снижение объёма, техника качества, восстановление, вес | добирать объём или начинать новые качества |
| 1-6 | свежесть, сон, вес, лёгкая техника | ЛМВ, тяжёлая сила, гликолитика |

### 7.2 Правила совместимости

Запрещено без явного тренерского подтверждения:

1. Тяжёлая ЛМВ ног + спринты на следующий день.
2. Тяжёлый хват + плотный партер/захват без восстановления.
3. Гликолитика + плохой сон + рост RHR.
4. Весогонка + добор интенсивности.
5. Чистая техника после случайного перегруза.
6. Тяжёлая сила в последние 7-10 дней.
7. Контактная борьба высокой плотности после локальной боли.

### 7.3 Правила выбора метода

```text
Если цель = скорость
и есть тесты 10/20 м
и взрыв нормальный
и нет боли ног,
то разрешить speed_first_action block.
```

```text
Если цель = ЛМВ ног
и есть боль колена/таза/спины,
то заменить на облегченный local_endurance_maintenance или технику.
```

```text
Если цель = техника под утомлением
но нет coachQualityScore,
то confidence не выше medium.
```

```text
Если до старта меньше 7 дней,
то development goals превращаются в maintenance/taper goals.

Если это главный старт и цикл 21-30 дней,
то календарная логика строится от эталона `docs/europe-2026-plan-analysis.md`:
вход в блок -> основной специальный микроцикл -> интеграция и снижение объёма -> подводка/пик.
Это не обычная "фаза развития".
```

## 8. Plan Structure

Итоговый план должен быть структурирован, а не свободным текстом.

```text
planId
athleteId
competitionId
generatedAt
cycleLengthDays
phaseMap
weeks
days
blocks
monitoringPoints
riskFlags
explanation
confidence
missingData
coachEditable
```

### Week

```text
weekNumber
phase
mainIntent
secondaryIntent
loadFocus
recoveryFocus
testOrControlPoint
```

### Day

```text
date
dayIntent
sessionCount
loadLevel
recoveryRequirement
readinessGate
blocks
generalComment
```

Активный тренировочный день должен собираться как эталонный план, а не как список изолированных стимулов:

```text
УТРО: разминка -> основная работа -> заминка
ВЕЧЕР: разминка -> основная работа -> заминка
```

Обязательная рамка дня:

1. `Разминка`: мобилизация, лёгкое движение, стойка/специальные движения, короткая активация без утомления в подводке.
2. Основная работа: СФП, борьба, техника, плотность, интервалы, хват, весовая или соревновательная задача.
3. `Заминка`: Z1/дыхание/мобилити, локальный сброс, контроль веса, сна и самочувствия.

Чисто служебный день контроля веса/сна без тренировочной задачи не нужно искусственно превращать в тренировку. Но любой активный день конструктора не должен выходить без разминки первым блоком и заминки последним блоком.

Если `sessionsPerDay = 2`, конструктор обязан сохранять две отдельные сессии: `УТРО` и `ВЕЧЕР`. Утром обычно ставится более физическая/энергетическая задача: СФП, сила, ЛМВ, скорость, интервалы, плотность. Вечером - технический перенос, борьба качества, контроль веса/сна, восстановление и подводящая активация. Это важно для аналитики: тренировка утром и тренировка вечером не должны сливаться в один общий список.

Каждый блок конструктора должен иметь не только общий смысл, но и упражнения. Недостаточно написать `2-4 подхода`: нужно дать конкретику, например `статодинамический присед 2-3x20-25 сек`, `входы в ноги 3x4`, `цикл 3 мин + 30 сек + 3 мин`, `канат 3x15-25 сек`, `мобилити 8-10 мин`. Тренер может редактировать объём, но черновик должен стартовать с рабочей конкретики.

### Block

```text
blockId
name
type
targetQuality
energySystem
localLoadZones
contactIntensity
volume
progressionStep
requiredData
riskFlags
evidenceRefs
coachEditable
```

## 9. Graceful Behavior при нехватке данных

Это ключевое поведение системы.

### 9.1 Нет теста под цель

Пример: выбрана скорость, но нет 10 м / 20 м.

Система делает:

- не строит агрессивный скоростной цикл;
- предлагает speed-maintenance или тестовую неделю;
- показывает `missingData: speed_tests`;
- confidence = low/medium;
- предлагает провести тест.

Текст для тренера:

```text
Для точного скоростного блока нужны 10 м и 20 м.
Сейчас система может предложить только осторожный блок поддержки скорости.
```

### 9.2 Нет readiness / сна / RHR

Система делает:

- не усиливает интенсивность;
- строит осторожный план;
- добавляет контрольные точки;
- просит готовность, сон, пульс покоя.

Текст:

```text
Вывод ограничен: нет данных восстановления.
Высокоинтенсивные блоки требуют подтверждения тренера.
```

### 9.3 Есть риск веса

Система делает:

- не добавляет гликолитику;
- не усиливает контакт;
- добавляет weight monitoring;
- просит комментарий тренера по весу/питанию/воде.

Текст:

```text
Вес требует контроля. Система не будет добирать объём, пока не понятен план снижения веса.
```

### 9.4 Есть боль в зоне нагрузки

Система делает:

- блок на эту зону помечает как риск;
- предлагает замену;
- требует подтверждение тренера.

Текст:

```text
Есть боль в зоне, на которую планируется нагрузка. Блок заменён на облегчённый вариант.
```

### 9.5 Данные часов неполные

Система делает:

- не считает часы абсолютной истиной;
- не затирает более полные данные;
- снижает confidence;
- просит дневник/комментарий.

Текст:

```text
Данные устройства неполные. План построен по доступным данным, но вывод по восстановлению ограничен.
```

## 10. Explanation Layer

Каждый план должен иметь `Почему так`.

Минимальная структура:

```text
Главное решение:
Почему выбран такой срок:
Какие тесты повлияли:
Какие риски ограничили план:
Какая доказательная база использована:
Что тренер может изменить:
Какие данные нужно обновить:
```

Пример:

```text
Главное решение: первые 2 недели развиваем скорость первого действия и ЛМВ ног, затем переносим это в технику.
Почему: до старта 42 дня, скорость ниже цели, взрывная база нормальная.
Ограничение: вес +1.4 кг и локальная усталость ног.
Риск: нельзя соединять тяжёлые ноги, спринты и плотный контакт без восстановления.
Что обновить: тест хвата, допустимую скорость снижения веса, тренерскую оценку качества входов.
```

## 11. Минимальный MVP

Первую версию конструктора не нужно делать полной. Достаточно:

### Вход

- соревнование;
- дата старта;
- спортсмен;
- цели;
- 5-8 ключевых тестов;
- состояние;
- ограничения.

### Выход

- план на 7/14/21/30 дней;
- недельная структура;
- блоки по дням;
- риски;
- `Почему так`;
- missing data;
- confidence;
- ручное редактирование тренером.

### Первые шаблоны

1. `pre_competition_21`.
2. `speed_first_action_14`.
3. `legs_lme_21`.
4. `arms_grip_21`.
5. `taper_10`.
6. `recovery_7`.

Эти шаблоны являются библиотекой методических якорей. Финальный черновик строится вторым шагом:

1. выбираются подходящие карточки;
2. извлекаются ключевые дни и блоки;
3. неделя расширяется до выбранной плотности `sessionsPerWeek`;
4. фаза недели считается от календаря подготовки, а не слепо наследуется из карточки;
5. недостающие дни заполняются низко- и среднеинтенсивной техникой, аэробной поддержкой, мобилити, контролем веса и сна;
6. тренер может редактировать каждый день и объём вручную.

При `sessionsPerWeek = 6` 21-дневный цикл не должен превращаться в 9-11 занятий, если только тренер явно не ограничил доступные дни или не включилась фаза восстановления. 10-дневная подводка не должна возвращать только `Д-10`, `Д-7`, `Д-3`; эти дни являются контрольными точками, а рядом должны быть короткие технические, восстановительные и активационные дни.

Календарное правило на базе Europe 2026 case:

- 30 дней: вход в блок -> вход/СФП -> основной специальный микроцикл -> интеграция/снижение объёма -> стартовое окно;
- 21 день: вход/основной специальный стимул -> интеграция -> подводка/пик;
- 14 дней: подводящий режим, если календарь соревнования уже поставил фазу `taper`;
- 10 дней: подводка + стартовое окно;
- 1-3 дня: только стартовое окно.

Карточка может содержать taper-неделю, но эта taper-фаза не должна появляться в середине 30-дневного цикла. Иначе система преждевременно уводит подготовку в сохранение свежести и теряет основной специальный стимул.

Дополнительные дни должны распределяться по всему стеку целей тренера. Если выбраны `speed_first_action`, `wrestling_contact_density`, `fatigue_skill`, `weight_management`, `taper_quality`, то черновик должен содержать не только скорость, но и плотность борьбы, технику под утомлением, контроль веса и качество подводки.

Если выбранные карточки развития не содержат taper-неделю, финальная подводка не должна переиспользовать развивающие дни. Например, при целях `legs_lme`, `arms_grip`, `aerobic_base` последняя неделя перед стартом не должна становиться ЛМВ/хватом с метками `Д-7`, `Д-6`; вместо этого строится отдельная подводящая неделя качества, восстановления и свежести.

В подводке нельзя строить день из одного изолированного блока скорости. Скорость первого действия в последние дни отображается как короткая нейромышечная активация и `taper_quality`, а не как развивающий `speed_first_action`. Такой день должен включать борцовский перенос: первое действие из стойки, входы, выход из захвата или тактическую ситуацию, затем короткую активацию и контроль веса/сна/свежести. Ускорения 20 м не могут быть всей тренировкой перед главным стартом.

## 12. Что нельзя делать в первой версии

1. Нельзя автоматически назначать план спортсмену без утверждения тренера.
2. Нельзя делать медицинские выводы.
3. Нельзя строить рискованный блок при низкой уверенности.
4. Нельзя скрывать нехватку данных.
5. Нельзя смешивать свободный текст с плановой структурой.
6. Нельзя считать данные часов абсолютными.
7. Нельзя строить “идеальный план” без учёта конкретного спортсмена.

## 13. Product Acceptance Criteria

Конструктор считается готовым для MVP, если:

- тренер может ввести старт, цели и спортсмена;
- система показывает, что она поняла;
- система показывает missing data;
- система строит редактируемый календарь;
- каждый ключевой блок имеет объяснение;
- рискованные сочетания подсвечиваются;
- при нехватке данных система ведёт себя осторожно;
- тренер может сохранить план как шаблон;
- ИИ-разбор потом понимает intent этого плана.

## 14. Следующий шаг

После утверждения этого документа:

1. Подключить UI-конструктор на сайте.
2. Показать draft, missing data, risks и `Почему так`.
3. Добавить редактирование дня/блока.
4. Добавить сохранение draft как шаблона.
5. После подтверждения тренером разрешить назначение спортсмену.

- `constructor-matrix-comparison.ts` — dual-run comparison layer для legacy vs matrix draft, safety invariants, legacy default guard и summary для internal preview.
- `constructor-matrix-preview.ts` — internal preview builder `buildConstructorComparisonPreview`, который возвращает side-by-side legacy draft, matrix draft, comparison report, summary, safety status, warnings и notes.
- `scripts/fixtures/constructor/preview-regression-fixtures.mjs` — synthetic regression fixture pack для matrix preview.
- `scripts/constructor-preview-fixture-runner.mjs` — fixture runner для safety, block, risk, explanation и legacy-default инвариантов.
- `POST /api/v1/plans/constructor/internal/matrix-preview` — internal/debug endpoint для matrix-vs-legacy preview с теми же auth/athlete-access guard'ами, что и production constructor draft route.

Старый `buildPerformConstructorDraft` не переключён по умолчанию.

### 15.1 Internal comparison preview

Preview-режим нужен для QA и будущей side-by-side панели, а не для production rollout.

Вызов:

```text
buildConstructorComparisonPreview(input, options?)
```

Output:

- `generatedFrom: "legacy_matrix_comparison_preview"`;
- `mode: "comparison_preview"`;
- `legacyDraft`;
- `matrixDraft`;
- `comparisonReport`;
- `summary`;
- `safety`;
- `safetyInvariants`;
- `legacyDefaultGuard`;
- `safeToPreview`;
- `defaultPathUnchanged`;
- `warnings`;
- `notes`.

Опции:

- `includeDrafts`;
- `includeComparisonReport`;
- `includeSafetyDetails`;
- `explanationDepth`;
- `failOnMatrixSafetyError`;
- `matrixOptions`;
- `includeInfoDifferences`.

Как читать:

- `safeToPreview = true` означает, что matrix draft можно показывать внутренне рядом с legacy;
- `expectedDifferenceCount` означает ожидаемые отличия, а не ошибку;
- `safety.matrixSafetyPassed` проверяет close-start, travel, weigh-in, competition, post-competition и marker rules;
- `safety.defaultPathUnchanged` подтверждает, что обычный legacy path не переключён;
- `warnings` показывает structured warning/error/info список.

На текущем этапе API, DB и UI не изменены. Preview не пишет в базу, не создаёт шаблон и не назначает план спортсмену.

Regression fixtures запускаются через `npm run check:constructor-core` и описаны в `docs/constructor-matrix-preview-fixtures.md`.

Обычный `POST /api/v1/plans/constructor/draft` не изменён. Internal endpoint принимает `{ input, options }`, возвращает preview object и не пишет в DB.

Следующий шаг — internal UI side-by-side panel или role-gated web preview, который вызывает internal endpoint.
